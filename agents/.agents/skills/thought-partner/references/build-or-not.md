Evaluate the idea against three gates. It passes all three or earns "do not build yet" with the smallest pre-build action.

Sequence gates blocking-first: one-pager → core tech → defining constraint. If evidence already exists for a gate, state provisional pass/fail and move to the weakest.

## Gate 1: One Page or It Doesn't Get Built

The idea needs a one-page north star: precise, ambitious, lean. Usable as source of truth for collaborators, investors, friends.

Empty space on the page = "research/prototype more." Overflow = "too complex."

Probe for: who it serves, what changes for them, what will **not** be built, why now, what conflict it resolves when stakeholders disagree.

## Gate 2: Core Tech Must Be Separable from the Product

A core method, tool, library, or skill that supports this product but **survives a product pivot**. All value trapped in the current product surface = it's a feature, not a compounding technical commitment.

Probe: is the core tech reusable IP, aligned with the builder's long-term vision, and valuable in at least one adjacent product?

## Gate 3: One Defining Constraint Must Shape the Product

A user-visible constraint that gives the product identity and reduces feature decisions. Users interact with it repeatedly. The product design "falls out" of it.

"Simple", "AI-powered", "fast", "easy to use" — not defining constraints. They create no identity, block no feature creep.

Probe for the Minecraft-blocks / IKEA-flat-pack equivalent: the rule users feel everywhere.

## NEVER

- **NEVER** let enthusiasm substitute for the one-pager. **Why:** excitement hides ambiguity until collaborators fight over scope. **Instead:** force the north star into one page or recommend research first.
- **NEVER** accept product features as core tech. **Why:** features die with pivots; core tech compounds across pivots. **Instead:** ask what survives if the product changes.
- **NEVER** accept invisible constraints as defining constraints. **Why:** internal rules give no user-felt identity and don't prevent feature creep. **Instead:** require a constraint the user repeatedly sees or is shaped by.
